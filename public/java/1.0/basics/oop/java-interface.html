Interface

Principe

  • Une interface est un ensemble de constantes et de déclarations de méthodes correspondant un peu à une classe abstraite.

  • Elle représente un contrat auquel une classe qui l’implémente doit répondre.

  • Tous les objets qui se conforment à cette interface (qui implémentent cette interface) possèdent les méthodes qui seront à redéfinir et les constantes déclarées dans celle-ci.

Mise en oeuvre

  • Une interface s’implémente dans une classe en utilisant le mot clé implements.

  • On peut implémenter autant d’interfaces que l’on veut dans une classe. (sorte d’héritage multiple)

  • Toutes les méthodes d’une interface sont abstraites : elles sont implicitement déclarées comme telles.

  • public or default access

  • non default methods assume to be public and abstract

  • variables are public static final

Polymorphisme

  • Une interface sert à définir un supertype et à utiliser le polymorphisme.

  • L’implémentation d’une interface définit un cast implicite de la classe fille vers cette interface..

Que se passe-t-il dans le cas d’une classe qui implémente 2 interfaces ayant une même méthode ?

  • 2 interfaces having the same method

    • if same signature and same return type OK

    • if different signature ⇒ overload

    • if same signature and different return type ⇒ KO

// 2 interfaces having the same method
// same signature and return type
public interface Herbivore {
    public void eatPlants();
}

public interface Omnivore {
    public void eatPlants();
    public void eatMeat();
}

public class Bear implements Herbivore, Omnivore {
    public void eatMeat() {
        System.out.println("Eat meat");
    }

    public void eatPlants() {
        System.out.println("Eat plants");
    }
}

// same signature and different return type
public interface Herbivore {
    public int eatPlants();
}

public interface Omnivore {
    public void eatPlants();
}

public class Bear implements Herbivore, Omnivore {
    public int eatPlants() { // DOES NOT COMPILE
        System.out.println("Eat plants");
        return 10;
    }

    public void eatPlants() { // DOES NOT COMPILE
        System.out.println("Eat plants");
    }
}

Interfaces VS Abstract Methods ?

  • a class can extend only from one abstract class, but a class can implement multiple interfaces

  • an abstract class can enforce a common state through instance variables (fields)

  • an interface can’t have instance variables

  • an interface extends an interface

  • an abstract class implements an interface (can implement several interfaces)

Default Methods (Java 8)

  • Les méthodes par défaut (default method) permettent de définir des méthodes, ainsi que leurs traitements dans une interface. Ceci permet à une classe qui implémente la méthode sans la redéfinir d’avoir le comportement de la méthode définie dans l’interface.

  • Elles permettent donc d’ajouter de nouvelles fonctionnalités aux interfaces en maintenant la compatibilité binaire avec les classes plus anciennes qui implémentent ces interfaces. Cela permet d’avoir une forme limitée d’héritage multiple dans les interfaces.

  • Problem of interfaces : when library designers need to update an interface to add a new method ⇒ need to be modified to reflect the new interface contract ⇒ new mechanism to tackle this problem

  • Interfaces can now have implementations :

    • static methods : not inherited in any classes ⇒ 2 interfaces with same static methods OK

    • default methods : allow you to provide a default implementation for methods ⇒ existing classes implementing an interface will automatically inherit the default implementations if they don’t provide one explicitly ⇒ evolve interfaces nonintrusively without breaking existing code (backwards compatibility)

      • examples : sort() and stream() in List interface

default void sort(Comparator<? super E> c) { Collections.sort(this, c);
List<Integer> numbers = Arrays.asList(3,5,1,2,6);
numbers.sort(Comparator.naturalOrder());

default Stream<E> stream() { return StreamSupport.stream(spliterator(), false); }

// useCase 1 : multiple inheritance behavior (see next slide)
// useCase 2 : Optional methods : classes that implement an interface but leave empty some method implementation (such as remove() when implementing Iterator because the user has decided to not use that capability) => unecessary boilerplate code
interface Iterator<T> {
	boolean hasNext();
	T next();
	default void remove() {
	throw new UnsupportedOperationException();
}

Default methods resolution rules

  • There is the possibility of a class inheriting more than one method with the same signature ! ><

  • How Java resolves this conflict (called diamond problem in C but more complicated in C because it accepts multiple class inheritance) ?

  • ⇒ 3 rules

    1. Classes always win. A method declaration in the class or superclass takes priority over any default method declaration

    2. Otherwise, sub-interfaces win : the method with the same signature in the most specific default-providing interface is selected (if B extends A, B is more specific than A)

    3. Finally, the choice is still ambiguous, the class inheriting from multiple interfaces hs to explicitly select which default method implementations to use by overriding it and calling the desired method explicitly ⇒ new syntax X.super.m with X the superinterface whose method m you want to call

Example 1 :
public interface A { default void hello() { sout(“Hello from A”); } }
public interface B extends A { default void hello() { sout(“Hello from B”);  } }
public Class C implements B,A { public static void main(String[] args  { new C().hello();  }} // What is get printed ?
=> Hello From B according to the second rule

Example 2 :
on rajoute une classe D telle que :
public class D implements A { }
public class C extends D implements B,A { … }
// Rule 1 says that a method declaration in the class takes priority but D does not override A => Hello From B according to the second rule

// BE CAREFUL : if D was declared like that :
public abstract class D implements A { public abstract void hello(); } => C would be forced to implement the hello() even though there are default methods elsewhere

Example 3 : B n’extends plus A
=> the Java compiler will produce a compile error because it doesn’t know which method is more suitable “Error: class C inherits unrelated defaults
for hello() from types B and A” => to resolve the conflict, you have to explicitly decide which method you want to use
public class C implements B, A {
	void hello() { B.super.hello(); }
}

Example 4 : Almost the same signature
public interface A { default Number getNumber() { return 42; }}
public interface B { default Integer getNumber() { return 42; }}
public class C implements B. A { psvm { sout(new C().getNumber()}; } // won’t compile as C can’t distinguish

Example 5 : Diamond problem
public interface A { default void hello() { sout(“Hello from A”); } }
public interface B extends A { }
public interface C extends A { }
public class D implements B, C {
	psvm { new D().hello(); }
} // => Hello from A as only A declares a default method, A is a superinterface of D

// if B has a default method => rule 2
// if B and C have a default method => rule 3 conflict => need to explicitly decide
// if C defines an abstract method void hello(); => rule 2 + need to redefine the method
