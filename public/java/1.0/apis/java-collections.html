API Collection

  • s’occupe de stocker et d’organiser des données en mémoirede façon efficace et performante

  • Opérations de base (analogie avec un sac) : ajouter, retirer, trouver, cardinal, itérer (pas d’ordre), taille extensible

Interface java.util.Collection<T>

  • add(T t), addAll(Collection<T> c) : ajoute tous les objets de la collection passée en paramètre à la collection dans laquelle je suis (opération d’UNION)

  • remove(Object o), removeAll(Collection c) : si on a une collection de String et qu’on demande à retirer un objet qui n’existe pas dans la collection, ce n’est pas une erreur ⇒ c’est pourquoi le paramètre est Object

  • contains(Object o), containsAll(Collection c)

  • size() : cardinal de la collection

  • clear() : effacer le contenu de la collection

  • retainAll(Collection c) : conserver les éléments dans laquelle je suis les éléments qui sont dans la collection passée en paramètre (opération d’intersection)

  • toArray()

Pattern Iterator

  • Iterator<String> it = c.iterator(); // interface Iterator

  • hasNext(), next(), remove() Iterator<String> it = c.iterator(); while (it.hasNext()) { String s = it.next(); if (s.length() > 5) { it.remove(); // si on avait fait c.remove(s) ⇒ exception (car je n’ai pas le droit de modifier une collection pendant que je suis en train d’iterer dessus) } }

Collection<String> = …​ al = new ArrayList<String>(); ll = new LinkedList<String>(); al.iterator() → Iterator<String> ll.iterator() → Iterator<String>

Pattern forEach

  • Iterable<T> forEach(Consumer<T> c)

  • Collection<T> removeIf(Predicate<T> p)

Types de collections

Il existe 3 interfaces qui étendent Iterable : * Queue * Set * List

Table 1. Types de collections
Type Description Implémentations Opérations

Queue

Collection d’objets où les éléments sont ordonnés pour le processing (FIFO, LIFO)

* LinkedList * ArrayDeque : more efficient than a LinkedList

* // Can not remove by index * boolean add(E e) // ajoute un élément à la fin et retourne true ou exception * E element() // retourne le prochain élément ou exception si vide * boolean offer(E e) // ajoute un élément à la fin et return true ou false * E remove() // enlève et retourne le prochain élément ou exception si vide * void push(E e) //(stack aussi) ajoute un élément au début * E poll() // enlève et retourne le prochain élément ou retourne null si vide * E peek() // retourne le prochain élément ou null si vide (stack aussi) * E pop() // (juste stack) enlève et retourne le prochain élément ou exception si vide

Set

* Collection non ordonnée d’objets uniques * Set n’ajoute pas de nouvelles méthodes mais change le comportement de Collection pour interdire les doublons

* HashSet * SortedSet (interface, extension de Set) * TreeSet (implémentation de SortedSet et de NavigableSet) * NavigableSet (interface qui étend SortedSet mais avec des features supplémentaires)

List

* Collection d’objets ordonnés = éléments accessibles par un index avec possibilité de doublons

* ArrayList * LinkedList

* get(index) * remove(index) * set(index, value) * add(index, value) * sublist(beginIndex, endIndex) // endIndex excluded * ListIterator<String> listIterator = list.listIterator(); hasPrevious(), previous() // on peut parcourir les éléments dans l’iodre croissant et l’ordre décroissant nextIndex(), previousIndex() // avoir la valeur de l’index * List<T> replaceAll(UnaryOperator<T> op) * List<T> sort(Comparator<T> op)

Implémentations de Set

Table 2. Extensions et Implémentations de Set
Type Description Implémentations Opérations

HashSet

Un Set avec une table de hash()

Comment HashSet sait qu’un élément existe déjà ou non dans le Set? * il va calculer le hashCode des objets et les comparer entre eux * si hashCode différent : les objets sont différents * sinon, appel à equals() pour vérifier s’ils sont bien égaux * optimisation car un hashCode est un entier facile à calculer alors qu’un equals est coûteux WARNING: ATTENTION A BIEN SURCHARGER HASHCODE ET EQUALS !

* Comment HashSet sait qu’un element existe deja ou non dans le Set ? Set<String> set = new HashSet(); set.add(“one”); // true set.add(“two”); // true set.add(“one”); // false

SortedSet (extension de Set)

* un Set trié

* Comment TreeSet conserve ses éléments triés ? ⇒ mécanisme Comparable / Comparator soit mes éléments sont comparables (implements Comparable) * présente une comparaison par défaut (ex: String par ordre alphabétique) * peut être surchargé par une implémentation de Comparator pour trier autrement soit mes éléments ne sont pas comparables ⇒ implémenter Comparator WARNING: trié (par ordre alphabétique) est différent d’ordonné (ordre d’insertion) ! Quand on itère sur une liste {b,c,d,a} ⇒ {b,c,d,a} Quand on itère sur un TreeSet {b,c,d,a} ⇒ {a,b,c,d}

NavigableSet (extension de SortedSet / Java 6)

Construire des listes pré-remplies

Table 4. Moyens pour construire des listes pré-remplies
Classe Caractéristiques Exemple

Arrays

  • classe Factory (= que des méthodes statiques) qui ne fait pas partie de l’API Collection

  • beaucoup de méthodes pour le traitement de données en mémoire, notamment sur les tableaux

  • Arrays.asList() : on ne peut pas modifier un élément (ni add ni remove) mais seulement remplacer (replace)

List<String> strings = Arrays.asList(“one”, “two”, “three”);

Collections

  • classe Factory

  • traitement de données à partir de collections dans la mémoire de la JVM

Set.of(), List.of()

  • Java 9+

  • immutables (on ne peut ni ajouter, supprimer ou modifier un élément)

List<String> strings = List.of(“one”, “two”, “three”);

Créer une liste à partir d’une autre

List<String> strings = List.of(“one”, “two”, “three”); List<String> stringsCopy = new ArrayList<>(strings); // copie de la liste qui ne modifie pas l’original et qui devient mutable

Classes obsolètes

  • Elles étaient déjà présentes (Java 1) avant l’API Collection (Java 2)

  • ArrayDeque, ArrayList et HashMap font leur jobs de façon plus performante

    • Vector : vieille implémentation d’ArrayList (thread safe)

    • Stack : ajout et retrait depuis le haut (extends Vector). Vieille implémentation de ArrayDeque

    • HashTable : vieille implémentation de HashMap

    • Dictionary

Collections concurrentes

Memory consistency : les 2 threads ne voient pas la même chose au niveau des données Blocking : méthodes d’attente avant d’effectuer l’action offer / poll / offerFirst / pollFirst / offerLast / pollLast Skip List : version concurrente de TreeSet et TreeMap CopyOnWrite : créer une nouvelle collection après chaque modification coût mémoire important
