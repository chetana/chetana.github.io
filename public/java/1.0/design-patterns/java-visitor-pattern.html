Pattern Visitor

Séparer les traitements du modèle objet

  • modèle objet = classes qui modélisent les business que notre app doit gérer

  • traitements = processus métier qui vont agir sur ces objets (ex: création d’un compte bancaire sur un objet Compte Bancaire)

Pourquoi vouloir séparer les traitements du modèle objet ?

les traitements sont des choses qui bougent + vite qui sont modifees sur une echelle de temps plus court que le modele objet ex: une app bancaire va gerer des clients, des comptes bancaires, des prets immobiliers = relativement constant dans le temps procedure d une creation de compte bancaire, d’obtention d un credit = varies avec le temps

on separe les 2 elements pour pouvoir livrer les traitements independamment du modele objet et de pouvoir faire des economies sur la livraison de l application dans sa globalite

Le pattern Visitor traite des hiérarchies d’objets.

Etude de cas

Company (référence un certain nombre de départements) 1n| Departement (référence un certain nombre de projets) 1n| Projet 1n| Employee

Caractéristiques de la hiérarchie : * modélisée par plusieurs interfaces (Company, Departement, Projet, Employee) * chaque interface peut avoir plusieurs implémentations projets financiers, informatiques département RH, juridique ** employés en CDD en CDI

On veut ajouter des features comme compter la moyenne des salaires des employées appartenant au pôle informatique.

Design du Pattern Visitor

interface Visitor : pour être capable l’intégralité des objets de la hiérarchie ⇒ il faut qu’elle ait autant de méthodes que d’objets

interface Visitor { void visitCompany(Company) void visitDepartement(Departement) void visitProjet(Projet) void visitEmployee(Employee) }

Dans chacune des interfaces de la hiérarchie, il faut une méthode qui prenne Visitor interface Company { void accept(Visitor v); }

interface Departement { void accept(Visitor v); }

Implémentation de la visite d’une hiérarchie d’objets

class HRDepartment implements Department { void accept(Visitor v) { // 1) transmettre v aux autres noeuds de la hiérarchie connus de Department ⇒ Projet this.projects.forEach(p → p.accept(v)); // 2) v visite l’objet this // callback : je recois un objet et j’appelle cet objet en retour en me passant en tant qu’instance // le visitor va me recevoir en tant qu’instance de HRDepartment et agir sur ce département v.visitDepartment(this); }

// coeur du pattern Visitor : on visite la hiérarchie et on appelle le visitor en callback = double dispatch

}

Implémentation d’un visiteur qui compte

class CountingVisitor implements Visitor { void visitCompany(Company c) { // rien } void visitDepartment(Department c) { // rien } void visitProject(Project c) { // rien }

    long count = 0L;
    void visitEmployee(Employee e) { count++; }
}

Bilan de ce pattern : * utile dans les hiérarchies d’objets de types différents modélisés par des interfaces / implémentations * intéressant lorsqu’il y a des traitements à faire sur ces objets (connus / futurs) code générique à ajouter aux éléments de ma hiérarchie (double dispatch accept(Visitor v)) ⇒ une des petites faiblesses du pattern car cela constitue une intrusion à l’intérieur des classes, ce qui peut être conséquent si on applique ce pattern sur un modèle existant ayant des centaines de classes * une interface unique à rajouter avec mon modèle objet, elle a une relation bidirectionnelle avec les différents objets de mon modèle (évolue en même tant que les objets) * les implémentations de Visitor sont séparés du modèle objet dans un autre module de traitement de données, elles dépendent des interfaces du modèle objet ⇒ mes traitements dépendent du modèle objet et peuvent bouger à la vitesse voulue et donc être livrées séparément avec un processus différent pour économiser le coût d’entretien de l’application
