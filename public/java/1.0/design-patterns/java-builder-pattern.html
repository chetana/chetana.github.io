Builder Pattern

Créer un objet qui possède beaucoup de variables au niveau des constructeurs. * Sert à construire des objets complexes = nombre de champs très importants problème de validité : règles de validation sur certains champs ou encore des règles qui vont prendre plusieurs champs en paramètres (si ce champ a telle valeur, l’autre ne peut pas) problème de complétude : construire des objets avec des champs manquants * Avoir beaucoup de variables devient rapidement lourd et illisible.

Moyens pour construire un objet

Constructeur : 3 problèmes : . 60 paramètres : error-prone . validation : si j’écris des règles de validation dans le constructeur je ne peux empêcher que la finalisation de la création de mon objet via une exception ⇒ antipattern . complétude : si je peux prévoir les champs manquants ⇒ je me retrouve avec plein de constructeurs différents qui auront des valeurs null selon la combinatoire voulue

Factory : . 60 paramètres : error-prone . validation : OK contrairement au constructeur, on peut faire des checks preliminaires . complétude : même problématique que le constructeur

  • Le builder pattern va résoudre ce problème en créant une classe inner qui va build l’objet. résoud les 3 problèmes évoqués ci-dessus 1) permet de gérer le grand nombre de params ⇒ je vais créer autant de méthodes que de paramètres avec le nom de la méthode correspondant au nom du paramètre 2) paramétres manquants : on n’appelle pas la méthode 3) la validation est gérée car la méthode build() est une factory

public class AnimalBuilder { private String species; private int age; private List<String> favoriteFoods;

public AnimalBuilder setAge(int age) {
    this.age = age;
    return this;
}
// ... pareil pour les deux autres variables
    public Animal build() {
        return new Animal(species, age, favoriteFoods);
    }
}
