Design d’Applications

Regroupe 3 éléments :

  1. Principes SOLID : le code qu’on écrit doit respecter un certain nombre de principes

  2. Design Patterns : ensemble de modèles de code qui permet de respecter les principes SOLID

  3. Architectures logicielles : qu’est-ce qui définit une architecture correcte ?

Architecture logicielle

Qualités d’une bonne architecture

  1. permettre de tester l’application et de la tester de façon automatique

    • essayer de faire fonctionner l’application dans un environnement qui essaie de reproduire au plus près l’environnement de production avec un jeu de données qui essaie de reproduire les données de production au plus près

    • objectif : détecter les défauts du code que l’on a écrit avant que ce code ne soit utilisé en production et cause des dommages sur la production de l’application

  2. permettre à cette application d’évoluer

    • une application logicielle vit en permanence et doit être modifiée pour 2 raisons (correction de bugs et besoins des clients qui évoluent)

  3. les choix techniques peuvent être revus : une bonne architecture est une architecture qui isole les choix techniques de façon à pouvoir les faire évoluer avec le temps et ne peut pas qu’elle dépende trop de ces choix

Principes SOLID

Exemple de José Paumard (cours-en-ligne sur Youtube)

Prenons une classe Employee qui contient 3 méthodes différentes : * saveToDB() : contient du code technique pour se connecter à la base et écrire en base * saveToXML() : écrit l’état (les champs) d’un employé dans un document XML destiné à un service Web * pay() : payer l’employé

public class Employee { saveToDB(); saveToXML(…​); // impacte le Web pay(); // impacte les RH }

Quel est le coût d’entretien de cette classe ?

  • Coût d’entretien = modification du code + livraison du package

    • On modifie le code, on le pushe sur un repo Git

    • Étapes jusqu’à la livraison (certaines peuvent être automatisées)

      • récupérer le code sur Git

      • compilation du code

      • tests unitaires

      • packaging de l’application

      • Q/A : tests manuels (nécessitent une intervention humaine)

      • Livraison : peut nécessiter une intervention humaine

Pour quelles raisons cette classe change-t-elle ? Quels sont les utilisateurs ?

  • saveToDB() : impacte les DBA

  • saveToXML() : impacte les acteurs du Web

  • pay() : impacte les RH

Quel est le coût et l’impact de la mise à jour de cette classe ?

  • Coût de mise à jour = coût d’entretien

  • Impact de la mise à jour sur les utilisateurs de l’application :

    • si les DBA demandent une modification, les autres sont impactés (interruption de service) même s’ils n’ont rien demandé car on doit recompiler toute la classe. Il peut même y avoir un risque de régressions !

    • pour limiter ces mises à jour :

      • on peut regrouper les livraisons, par exemple une par mois mais si les DBA demandent une feature le 2 du mois, ils ne pourront l’avoir livrée que le 30

      • automatiser la mise à jour ⇒ il faut annuler le nombre de personnes qui ont un impact sur les modifs de cette classe (idéalement 0 pour ne plus jamais avoir de raison de changer de classe mais plutôt 1 dans la réalité)

Single Responsibility Principle / Responsabilité d’une classe

  • Nombre de responsabilité d’une classe = nombre d’acteurs qui peuvent demander des modifications dans cette classe (dans l’exemple ci-dessus, 3 (éventuellement +1)) = nombre de raisons de changer la classe

  • Single Responsibility Principle (correspond au principe S du SOLID) : on ne doit avoir qu’une seule raison de changer un bloc de code, une methode, un module, une classe et une raison de modification doit être une raison métier (analyst business, métier…​)

Cela ne veut pas dire que la classe doit seulement faire une seule chose ! Le nombre de responsabilités et le nombre de fonctionnalités sont des notions differentes.
Diminuer les responsabilités en utilisant la délégation

Délégation : déléguer une tâche technique à un objet tiers * si je reprends saveToDB(), je dois enlever le code technique permettant de se connecter et écrire à la BDD DBService dbService; saveToDB() { dbService.saveToDB(this); }

  • La classe Employee dépend de la classe DBService à la compilation

  • Maintenant, quand un DBA me demande une mise à jour, je n’ai plus besoin de modifier la classe Employee, seulement la classe DBService ⇒ le nombre de responsabilités de la classe Employee a bien été diminué

  • Mais cela ne règle pas les impacts sur les autres utilisateurs car quand je recompile DBService, il faut également recompiler et passer par toutes les étapes de la livraison pour la classe Employee.

  • Le deuxième problème qui se pose est cette dépendance. Comment inverser cette dépendance ?

Dependency Inversion Principle

  • Dependency Inversion Principle (correspond au principe D du SOLID) : ce sont les détails d’implémentation qui doivent dépendre des notions de haut niveau quand je veux inverser la dépendance entre 2 modules dans une aplication, il suffit que j’insère une interface entre les 2 et ce faisant les dependances a la compilation vont toutes les 2 pointer vers cette interface. le fait de mettre une interface ca permet de proteger le code client contre les modifications des implementations. d une facon generale les dependances au niveau du code doivent TOUJOURS aller des regles d implementation vers les regles de haut niveau.

  • dans l’exemple :

    • ma notion de haut niveau = enregistrer mes employés dans la BDD

    • le détail d’implémentation = ma BDD est une BDD MySQL

  • Quand on veut changer un détail d’implémentation, cela n’a d’impact sur aucune autre partie de l’application (par exemple passer sur une BDD Oracle)

Inverser une dépendance à l’aide d’une interface

Interface : utiliser une interface pour inverser une dépendance * Employee dépend de DBService en tant qu’interface * Il faut une implémentation de DBService comme MySQLDBService (me permet de me connecter à une base MySQL) * L’implémentation dépend de l’interface * Je vais déployer Employee et DBService dans un module et MySQLDBService dans un autre module ⇒ j’aurai MySQLDBService qui dépend d’Employee et de DBService et non l’inverse * si je dois changer la logique de connexion de la BDD, je n’ai qu’à mettre à jour MySQLDBService et cela n’aura pas d’impact sur l’autre module ⇒ livraison moins chère et surtout qui n’impacte pas les autres utilisateurs

Comment instancier ce service et la récupérer dans Employee ?

Injection de dépendances public Employee(DBService dbService) { this.dbService = dbService; }

Organisation des modules de l’application (cf image)

  • Le module Employee encapsule les règles métiers de haut niveau.

  • Les autres modules sont les détails d’implémentations qui dépendent du module de haut niveau.

  • Le module Main

    • il s’agit du module de plus bas niveau

    • il va se charger de construire les différentes implémentations des interfaces puis de construire un objet Employee en lui injectant les instances de ces interfaces

la classe Employee ne devrait comporter que du code concernant son état (= le modèle) et pas des méthodes saveToDB()…​ car on a une relation circulaire entre Employee et les interfaces alors que seules les interfaces ne devraient dépendre d’Employee

Barbara Liskov Substitution Principle

Revenons sur la notion d’héritage. On dit souvent qu’il y a une notion d’héritage "naturelle" avec la relation "est un" mais c’est un petit peu plus complexe que ça.

Exemple 1 : un nombre entier est un nombre réel

  • comment sont encodés chacun d’entre eux ?

    • nombre entier : encodé sur 32 bits via le complément à 2

    • nombre réel : encodé sur 32 bits via le nombre à virgule flottante

  • quel est le rapport entre les 2 encodages ?

    • AUCUN

Integer et Float est un exemple d’une relation de type "est un" dans le monde réel mais qui ne se translate pas vers une relation d’héritage dans une application logicielle dû à l’encodage !
Exemple 2 : un nombre réel est un nombre complexe

public class Float extends Complex {} public class Complex { Float r, Float im // partie réelle, imaginaire // Constructeur Complex(Float r, Float im) {} } * On voit qu’il y a une relation bidirectionnelle entre Float et Complex : Float étend Complex : pour construire un Float j’ai besoin d’un Complex pour construire un Complex, j’ai besoin de 2 Float ** chacun appelle les constructeurs de l’autre ⇒ StackOverflowException

cette relation bidirectionnelle est un autre exemple de relation "est" qui ne mène pas à une relation d’héritage
Exemple 3 : un carré est un rectangle particulier

public Rectangle { int width; int height; } public Square extends Rectangle { int size;

setSize(size) {
    setWidth(size);
    setHeight(size);
}
// or Rectangle a des setter setWidth() et setHeight() qu'il faut que Square surchage pour modifier la valeur de height quand width est settee et réciproquement
setWidth() {}
setHeight() {}
// mais le setSize() doit appeler le setWidth() de Rectangle et non celui surchargé
// 1er problème dû aux règles de la programmation objet et la façon dont l'héritage et la surcharge fonctionnent
    // il y a aussi des pbs au niveau du comportement
    // une des props du rectangle est si j'augmente une de ses dimensions d'un certain montant, alors sa surface augmente dans les mêmes proportions (si j'augmente de 20% la largeur => la surface aussi)
    // MAIS ce n'est pas vrai pour un carré !
}
le carré ne se comporte pas comme un rectangle ⇒ dans la classe Rectangle, j’ai une notion d’invariance : je peux modifier la largeur mais cela ne modifie pas la hauteur. Définir un invariant = quand je touche une propriété, les autres ne sont pas modifiées ⇒ aucun moyen en Java de dire ça. Est-ce qu’un carré se comporte de la même façon qu’un Rectangle = non !
Conclusion sur la notion d’héritage

  • L’Héritage n’est pas juste une relation de type "est" mais AUSSI une relation de type "se comporte comme"!

  • Si A "se comporte comme" B alors A étend B

Comment détecter dans du code qu’on crée de l’héritage entre des classes qui ne se comportent pas de la même façon ?

On est obligés d’ajouter du code dans la classe fille pour revoir certains comportements de la classe parente ⇒ pas de notion d’héritage "naturelle"

Liskov Substitution Principle

Liskov Substitution Principle (correspond au principe L du SOLID) :explique ce qu’est l’héritage en programmation objet Si A "se comporte comme" B alors A étend B, alors partout dans mon application où j’utilise des instances de B, je peux utiliser des instances de A de la même façon. si l’instance d’une classe B peut etre utilisee a la place de l instance d une classe A sans que le processus qui utilise ces instances ne voit de differences, cela veut dire que B etend A * Parmi les 3 exemples précédents : ex1: je ne peux pas utiliser les nombres entier et réel de la même façon (différent mécanisme au niveau de l’arithmétique) ex2: je ne peux pas utiliser un nombre réel comme un nombre complexe (je n’arrive même pas à le construire) ** ex3: je ne peux pas utiliser un carré comme un rectangle, si j’augmente la largeur d’un carré de 20% la surface de ce carré n’augmente pas de 20% ⇒ le principe de Liskov est violé !

Etude de cas pour appliquer S,L,D

Contexte

  • Une entreprise qui veut créer une application pour gérer ses salariés

  • Ils sont séparés en 2 catégories : CDD et CDI (qui ont des règles juridiques différentes)

  • Les processus métiers attachés aux employés : relatifs à la DB et les autres au versement de salaire pay()

Que dit S ?

  • Le principe S nout dit qu’il y a des règles différentes pour les CDI et les CDD et donc des raisons différentes et indépendantes de modification de ces règles

    • un module pour les CDI et un module pour les CDD

    • un employé en CDI modélisé par une classe et un CDD par une autre classe

Que dit L ?

  • Dans notre application on a un certain nombre de processus metiers qui gèrent les employés CDI et les CDD

  • L’execution et le declenchement de ces processus metiers ne doivent pas dependre du fait qu’un employe est en CDI ou en CDD ex: si je dois lire un employe dans la BDD, la commande qui permet d’aller lire cet employe ne depend pas du fait qu il soit en CDI ou en CDD, ce qui va dépendre de ca ce sont les details d implementations qui ne doivent pas apparaitre dans le code primaire

  • Cela impose d’avoir une structure au dessus de CDI et de CDD qui va être la structure manipulée par l’application ⇒ Employee

Que dit D ?

  • Cette structure de haut niveau Employee ne doit pas dépend pas de CDI et CDD mais ce sont eux qui dépendent d’Employee (structure de haut niveau) (principe D)

  • Dois-je utiliser Employee en tant qu’interface ou classe abstraite ?

    • la convetion est d’utiliser une interface car une interface c’est ce qui modelise ce que doit faire une classe

Organisation de l’application (cf image)

  • 2 classes concrètes CDD et CDI qui implémentent l’interface Employee

  • Le module Application qui consomme cette interface pour appeler les différntes règles et processus métiers relatifs à Employee (déclenchement de sauvegarde dans la BDD, déclenchement du processus de paiement)

Comment créer des employés ? ⇒ Pattern Factory

EmployeeFactory createCDI(…​) createCDD(…​) * S’agit-il d’une classe ou d’une interface ? Classe : createCDI() { return new CDI(); } * cette classe va dépendre des implementations de mon application (cf image) * cette classe est appelée par le module Application qui se retouve de manière transitive dépendante des implémentations ⇒ KO Interface : EmployeeFactory Employee createCDD() Employee createCDI() * EmployeeFactoryImpl = l’implémentation de l’interface EmployeeFactory qui dépend de CDD et CDI mais on a quand même une couche d’isolation grâce à l’interface entre les notions de haut niveau et les détails d’implémentation qui sont en dépendance de ces règles de haut niveau. (cf image)

Organisation de l’application à l’exécution

  • A la compilation, Application ne depend que des interfaces (EmployeeFactory f;)

  • c’est a l execution que les interfaces seront implementees a partir de la methode main() = seul module a avoir l integralite des modules de mon application en dependance (haut niveau + details d implementation) (new EmployeeFactoryImpl()…​) WARNING: injection de dependances, c est ce qui se passe a l execution et inversion de dependances c est le principe qu on utilise lorsqu on concoit les differents modules d une app

Ajout d’une implémentation pour Employee

Si on ajoute une 3ème implémentation Contractor : * à la compilation pas de problème * mais d’un point de vue design, il va quand même falloir ajouter une méthode createContractor() dans EmployeeFactory qui dépend du coup des détails d’implémentations * Comment casser cette ultime dépendance et éviter de rajouter du code dans ma factory à chaque ajout d’une implémentation? ⇒ passer d’autant de méthode de création d’implémentations à une seule !

Pattern Abstract Factory

EmployeeFactory : Employee create(String label, …​) { // label va me permettre de savoir quelle implémentation il faut créer switch(label) { case "CDD": return new CDD(); case "CDI": return new CDI(); } } WARNING: l’utilisation d’un switch est déconseillé car susceptible de se dupliquer dans plein d dendroits ou je vai devoir faire des traitments anoalogues a faire sur ce meme genre de criteres on peut en avoir plusieurs partout dans le code et à chaque ajout d’implémentation, il faut modifier les switch mais on peut en oublier !

Pattern Strategy

  • extrêmement important car utilise partout dans les app info de gestion

  • partout on va switcher, faire des if et else sur un certain nombre de critere pour appliquer un certain nombre de traitements, pour modifier des donnees d une certaine facon

  • Strategy = traitement que j’applique en fonction d’un certain critere

    • ici : new CDD(), new CDI(), new Contractor() ⇒ correspond à une branche de code

  • Critère = permet de decider quelle strategie va etre appliquee

    • si label vaut CDD, CDI, Contractor …​

  • Modèle de stratégie : entrées/sorties de notre stratégie ⇒ il faut trouve un jeu de paramètres communs (return new CDD() ⇒ on crée un objet Employee avec une liste de paramètres…​)

Création des classes du pattern Strategy

  • création d’une classe abstraite

    • AbstractEmployeeType abstract Employee create(…​) // le modèle de la stratégie permet de déduire la singature de cette méthode

      • Implémentation CDDType Employee create(…​) { return new CDD(…​); }

      • Implémentation CDIType Employee create(…​) { return new CDI(…​); }

  • j’ai échangé le switch contre une structure avec une classe abstraite qui porte le modèle de ma stratégie et autant de classes concrètes que j’ai à l’intérieur de ce switch qui vont fournir une implemetnation concrete de la methode abtraite

Code Client

AbstractEmployeeType type = AbstractEmployeeType.of(type); Employee employee = type.create(…​);

Open Closed Principle

Open Closed Principle (correspond au principe O du SOLID) : si je veux évoluer les features de mon application, la seule chose que j’ai besoin de faire c’est d’étendre certaines classes bien choisies de cette application le code client est fermé à la modification et que si je veux augmenter le nombre de features de mon application, je dois le faire en étendant des classes abstraites, en fournissant des implémentations supplémentaires (ouvert à l’extension) sans toucher au code client

Pattern Strategy par composition

Actuellement on a : public class CDDType extends AbstractEmployeeType { // stratégie public Employee create(String name) { return new CDD(name); } }

On peut implémenter ce pattern différemment en utilisant la composition, c’est-à-dire une classe unique que l’on peut instancier en passant des lambdas expressions vers lesquels on va deleguer les differentes operations de notre strategie. pour simplifier le pattern strategy. ⇒ on va encoder la stratégie dans une lamda expression

public EmployeeType { String label; // CDD, CDI, Contractor Function<String, Employee> create; // la stratégie

EmployeeType(String label, Function<String, Employee> create) {}
    String type() { return label;}
    // va déléguer la création à la lambda
    // plutôt que d'avoir autant de méthodes create() que j'ai de types d'extensions dans ma classe abstraite
    // j'ai une seule méthode create et autant d'instances de la classe concrète EmployeeType que j'ai de types
    Employee create(String name) {
        return create.apply(name);
    }
}

EmployeeType cdd = new EmployeeType("cdd", name → new CDD(name)); EmployeeType cdi = new EmployeeType("cdi", name → new CDI(name)); List<EmployeeType> types = List.of(cdd, cdi); Employee create(String label, String name) { return types.stream() .filter(type → type.label == label) .findAny() .map(eType → eType.create(name)) .orElseThrow(); }

Bilan : 2 façons d’implémenter le pattern strategy 1) la façon canonique expliquée dans le GoF : consiste à avoir une classe abstraite et des classes concrètes 2) par composition : une classe unique avec un nombre d’instance bien déterminées et qui délègue les stratégies à des lambdas expressions modélisées dans cette classe

Interface Segragation Principle

Interface Segragation Principle (correspond au principe I du SOLID) : je ne dois pas dépendre d’une interface qui possède des méthodes dont je n’ai pas besoin = je dépends d’interfaces et dans ces interfaces, j’ai le nombre minimum de méthodes dans ces interfaces pour fonctionner quand un service, une classe, un processus definit une dependance vis a vis d une interface, si possible les methodes de cette interface doivent etre minimales par rapport aux methodes dont ce client a besoin. un client qui dependrait d’une interface et qui ne dependrait pas de toutes les methodes de cette interface s’expose a des pbs quand ces methodes non utilisees devront evoluer. le pb principal est qu;il va devoir se reconstruire et payer le cout de cette reconstruction de facon inutile

ex: j’ai une classe InvoiceProcess qui dépend d’une interface VATService et qui utilise sa méthode getVAT() interface VATService { int getVAT(String countryCode); }

class InvoiceProcess { VATService service; InvoiceProcess(VATService s) { this.service = s; }

    int computeVAT(int amount) {
        return service.getVAT("FR") * amount;
    }
}

Or, à un moment l’équipe en charge de l’interface VATService ajoute des méthodes pour leurs besoin dont notre code client InvoiceProcess n’a pas besoin. ⇒ nous force à recompiler de notre côté et à repasser par toutes les étapes de livraison. De plus, il y a eu une erreur dans la typo de nom d’une méthode ⇒ ils recompilent ce qui nous force encore à faire de même chez nous. ⇒ ils auraient dû au niveau du code de l’application créer une nouvelle interface qui serait implémentée par l’implémentation de VATService, ce qui aurait simplifié les choses au niveau du code client

interface VATService { int getVAT(String countryCode); // Méthodes ajoutées non utilisées par InvoiceProcess int setVAT(String countryCode); CountryCode getCountriCode(String countryCode); }

Architecture d’une application

Comment organiser les modules d’une application et ce que l’on va mettre dans chacun de ces modules ? (Voir ça comme des cercles concentriques cf image)

  1. Mettre au centre des applications ce qui bouge le moins souvent = Modèle objet ex: pour une app voyage, le modèle objet sera constitué d’objets Voyage, Avion, Chambres et cela va peu bouger dans la vie de l’application

  2. Traitements métiers : ils dépendent du modèle objet mais pas l’inverse pour ne pas avoir à livrer le modèle objet à chque fois qu’il y a une modification du traitement métier qui bouge pluss! Ces traitements métiers vont avoir besoin de données venant de l’extérieur :

    • IHM : natives, mobiles ou web

    • BDD : relationnelles ou no SQL

    • Services : pouvant fournir des données ou d’activer des traitements sur d’autres applications ou d’autres parties de notre application (SOAP ou REST)

  3. Tout ça est modélisé par des Controller et Presenter (IHM) ou Gateway (accès réseau) De même, ce sont eux qui dépendent des traitements métiers et non l’inverse !

Implémentation IHM Flow of control (ce qui se passe à l’exécution)

(cf image) Pour présenter les traitements métiers à l’utilisateur ⇒ module Presenter Ce module Presenter va présenter le contract sur une Vue L’utilisateur va updater des champs via un Controller qui update les traitements métiers

ce flow of control est normal à l’exécution mais les dépendances à la compilation ne doivent pas etere comme ca

  • Le Presenter doit dépendre du module de MAJ Contrat ⇒ principe D on rajoute une interface Output

    • De plus, la MAJ des contrats doit transmettre des données au Presenter ⇒ en passant un objet Contrat mais sous forme d’un objet de transport = DTO (Data Transport Object) = objet indépendant du module Contrat (modèle objet), il est créé par le processus métier (local à MAJ Contrat)

    • Le module du Presenter et Controller va également avoir son propre objet transport à donner à la Vue qui ne doit pas dépendre du processus métier (et ne contient souvent que des Strings et nombres ce qui fait que l’objet transporté est plus léger)

Implémentation de l’accès à la base de données

BDD relationnelles ⇒ Hibernate / JPA De même, pour respecter les principes SOLID il faudrait qu’Hibernate dépende de notre modèle objet NOTE: dans 90% le modèle objet porte les annotations Hibernate ⇒ ce schèma n’est pas du tout respecté !
