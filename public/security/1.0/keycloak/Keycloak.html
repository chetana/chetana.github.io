Keycloak

KeyCloak est un serveur de gestion d’identité et d’authentification.

Pourquoi déléguer sa sécurité ?

  1. Par souci d’économies d’échelle

    • le faire soi-même requiert de développer :

      • écran de login

      • gestion du workflow

      • gestion des mots de passe

      • gestion des nouveaux utilisateurs

  2. Vous n’êtes pas expert en sécurité

    • On peut penser a faire un serveur pour centraliser toute la gestion de login mais cela ne suffira sûrement pas (cf anecdote avec T-Mobile Australia : cette société se la jouait mais elle ne hachait pas ses MDP)

Qu’est-ce que Keycloak ?

  • Surcouche de WildFly (serveur d’applications)

  • Développé par RedHat

  • Keycloak repose en partie sur des normes et protocoles de sécurité déjà existants (OpenID)

    • Avantage : on se protège des trous de sécurité et on a une intéropérabilité avec les autres composants du marché.

Fonctionnalités

Signature

  • Keycloak crée une paire clé publique/privée

  • Il signe avec sa clé privée

  • l’application dans le même realm demande la clé publique à Keycloak

    • Avantage : permet à Keycloak de renouveler ses clés

Protection des attaques

  • Brute force detection : bloquer le compte après trop de tentatives infructueuses

  • Clickjacking

  • Injection SQL

  • Les MDP sont hashés avec PBKDF2 (20 000 itérations)

Absolument se mettre en HTTPS pour protéger de la majorité des attaques !

Persistence

  • Hibernate derrière

  • Supporte Postgre, MySQL

Clustering / Cross Data Center

Concepts

Realm

  • on va donner un "territoire" à sécuriser à Keycloak

  • un territoire peut contenir plusieurs applications et est isolé des autres

  • on peut faire du SSO sur un realm

  • Analogie : on prend une carte avec plusieurs territoires. on navigue vers un territoire et celui-ci nous redirige vers l’ile de Keycloak dans laquelle on doit fournir nos credentials (login form). Une fois renseignés, on obtient un tampon de Keycloak et on peut aller visiter le territoire.

Console d’administration (interface web) Keycloak

  • toutes les opérations de la console sont basées sur l’API ⇒ possibilité d’automatiser les opérations

  • Opérations possibles :

    • Création d’un realm

    • Création d’un client

    • Création d’un utilisateur + mdp (on peut forcer l’utilisateur à changer son mdp lors de sa première connexion)

    • Customisation de la page de login dans "Realm Settings"

    • Configuration OTP

    • Défense du territoire

  • Applications fournies par Keycloak :

    • Account : gestion du compte utilisateur

Adaptateurs

  • Sécurisent les applications et communiquent avec Keycloak

  • S’intègrent au système de sécurité existant

  • Configuration standardisée [source, json] ==== { // où tourne le serveur Keycloak "auth-server-url": "http://localhost:8080/auth", "realm": "realm-name", "resource": "client-id", // application qui ne peut plus recevoir des requêtes avec un token dejà existant "bearer-only": true, "ssl-required": external, "enable-cors": true } ====

  • Que font-ils?

    • Gestion des Redirect (ou pas si c’est un bearer-only)

    • Vérification des signatures : avoir besoin de la clé publique pour vérifier la signature du token

    • Vérification des Claims : vérification du payload de notre token

  • Workflow : Requête entrante

    • le path est-il soumis à une contrainte de sécurité ? Y a-t-il un header "Authorization"?

      • Cas 1: header absent

        • Le client est-il confidential ou public?

          • Echange eventuel du secret puis la redirection vers la page de login est initiee

          • Le client est bearer-only ⇒ unauthorized (ex: un service REST avec lequel il n’y a pas d’intéraction humaine, le rediriger vers la page de login n’a pas de sens)

      • Cas 2: header présent

        • la signature est-elle valide?

        • le token est-il expiré (exp)?

        • vérification issuer (iss)

        • vérification audience (aud)

        • Matching éventuel du not-before

  • Adaptateurs possibles :

    • Java EE / Servlet container

    • Spring Boot / Spring Security

    • NodeJS

    • Javascript

    • Servlet Filter

    • Proxy

Security Defenses

  • Brute force detection :

    • quelqu’un qui bombarde le serveur en testant le dictionnaire des mdp et espère tomber sur le bon ⇒ on peut locker les comptes

    • un problème avec cette défense : quelqu’un peut utiliser le même mécanisme pour locker tous les comptes de vos users

    • la log est parsable par fail to ban : locker les gens qui font trop d erreurs de mdp

Clients

  • 3 types d’accès au client :

    • public :

      • la sécurité du client public repose uniquement sur l’url de redirection (c’est l’url qui va récupérer le code et l’échanger contre le token sans s’identifier)

      • pour des applications single page ou mobile

    • confidential :

      • on sécurise l’échange avec des secrets/credentials

      • pour des applications server side

  • bearer-only :

    • application qui ne peut plus recevoir des requêtes avec un token dejà existant

  • La partie Installation permet d avoir les configurations du client que l’on peut mettre dans l’application.yml du Springboot.

Exemples d’utilisations

Sécuriser une application Spring Boot

  • On peut partir d’un projet start.spring.io pour les dépendances

  • Pour intégrer Keycloak avec Spring Boot, il existe 2 adapters différents :

    1. le premier situé juste au niveau de SpringBoot et sécurise le serveur web utilisé par Springboot (Tomcat, Undertow, Jetty)

    2. l’autre se plug sur Spring Security

configuration de Keycloack dans l’application.yml
keycloak:
    # URL du serveur d'authentification
    auth-server-url: http://localhost:8080/auth
    # territoire
    realm: devoxx
    # identifiant de l'application = client à déclarer dans Keycloak
    resource: beer-boot-app
    # correspond à ce qu'on peut afficher en tant qu'identifiant de Keycloak (nom, uuid...)
    principal-attribute: name
    # si on a défini l'access type du client à confidential, on a besoin du secret
    credentials:
        secret: 0e40556a-0563-XXXX
